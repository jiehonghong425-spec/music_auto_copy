# 人声分离功能优化报告

> 项目：AudioJungle 预览音频批量下载 + 人声分离  
> 日期：2026-07-20  
> 引擎：Meta Demucs v4.1.0

---

## 一、问题诊断

### 初始状态

人声分离代码已完整编写（`src/separator/local_engine.py`，362 行），但实际运行**全部失败**：数据库 20 首音乐中 1 首 `failed`、1 首卡在 `separating`、`separated` 目录下全是空文件夹。

### 根因分析

| 问题 | 严重度 | 说明 |
|------|--------|------|
| **FFmpeg 缺失** | 🔴 致命 | Demucs 内置 torchaudio 解码部分 MP3 时抛出 `malformed stream: mpa: invalid main_data offset`，需要 FFmpeg 作为后备解码器，但系统未安装 |
| **错误静默吞没** | 🟡 中等 | `_run_demucs()` 返回 `False` 但不记录具体错误，stderr 被丢弃 |
| **残留清理缺失** | 🟡 中等 | 失败后 `_tmp` 目录残留，`separating` 状态卡住不恢复 |
| **性能低下** | 🟠 较高 | `htdemucs_ft` 是 4 个模型的组合（bag of 4），CPU 串行处理，每首 ~5 分钟 |

---

## 二、修复方案

### 2.1 安装 FFmpeg

```bash
winget install Gyan.FFmpeg
```

安装版本：FFmpeg 8.1.2（通过 WinGet）

### 2.2 代码修复（`src/separator/local_engine.py`）

| 修改项 | 修改前 | 修改后 |
|--------|--------|--------|
| **FFmpeg 检测** | 无 | 新增 `_find_ffmpeg()` 函数，自动搜索常见安装路径并加入 PATH |
| **启动检查** | 仅检查 demucs | `_init_separator()` 先检查 FFmpeg，缺失则报明确错误并给出安装指引 |
| **错误日志** | `_run_demucs()` 返回 `bool`，失败无详情 | 返回 `tuple[bool, str]`，提取 stderr 关键错误行 |
| **失败清理** | 无 | `_separate_one()` 失败时清理 `_tmp` 残留目录 |
| **成功日志** | 无 | 输出文件大小信息（`✓ 标题 (伴奏: XKB, 人声: YKB)`） |
| **参数兼容性** | `--two-stems=vocals` | `--two-stems vocals`（空格分隔，兼容性更好） |

### 2.3 性能优化

| 优化项 | 修改前 | 修改后 | 提升 |
|--------|--------|--------|------|
| 模型 | `htdemucs_ft`（bag of 4） | `htdemucs`（bag of 2） | 子模型数减半 |
| CPU 并行 | 无 | `-j 4`（利用 16 核 CPU） | 多核并行 |
| **单首耗时** | **~300 秒 (5 分钟)** | **~107 秒 (1.8 分钟)** | **≈ 3× 加速** |

模型选择对比：

| 模型 | 子模型数 | 单首耗时 | 质量 |
|------|----------|----------|------|
| `htdemucs_ft` | 4 | ~300s | 最高 |
| `htdemucs` | 2 | ~107s | 优秀 |
| `mdx_extra` | 7 | ~392s | 一般（且更慢） |

最终选择 `htdemucs -j 4`，在质量和速度之间取得最佳平衡。

### 2.4 配套修改

- **`config.yaml`**：默认模型改为 `htdemucs`
- **`requirements.txt`**：新增 FFmpeg 安装说明
- **`separate_one.py`**：新建简化的单首处理脚本，支持断点续传，适用于批量处理和定时任务

## 四、经验教训

1. **FFmpeg 是 Demucs 的隐性依赖**：虽然 demucs 的 `requirements.txt` 不包含 FFmpeg，但实际运行中部分 MP3 文件必须通过 FFmpeg 解码。应在项目初始化时显式检查。

2. **非 NVIDIA GPU 环境需注意**：本项目运行在 AMD 集成显卡 + CPU-only PyTorch 环境。`torch-directml` 理论上可启用 AMD GPU 加速，但 512MB 显存不足以加载 Demucs 模型。CPU 16 核 + `-j 4` 是目前最优方案。

3. **子进程错误应透传**：`subprocess.run(capture_output=True)` 后若仅返回 `bool`，排查问题极其困难。必须将 stderr 关键信息写入日志或数据库。

4. **后台任务超时管理**：CLI 工具的超时机制会终止长时运行的后台任务。对于批量处理场景，应采用「逐首处理 + 断点续传」的架构，而非单次长时任务。
